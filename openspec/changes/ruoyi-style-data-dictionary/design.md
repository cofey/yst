## Context

当前后端已落地若依风格 RBAC（菜单、按钮、`@PreAuthorize("@ss.hasPermi(...)")`）与前端按钮权限指令，但缺少“数据字典”统一能力。现状是页面字段状态、选项值更多依赖硬编码，无法通过后台配置统一维护。  
本次设计要求前后端均遵循若依字典模式，确保模型、接口、权限点和页面交互方式一致，并与现有 `system:*:*` 权限体系兼容。

## Goals / Non-Goals

**Goals:**
- 提供若依式字典模型：`sys_dict_type` + `sys_dict_data`。
- 提供字典类型与字典数据管理接口，接入现有权限体系（`system:dict:*`）。
- 提供按 `dictType` 查询字典数据的前端消费链路，避免页面硬编码。
- 建立字典缓存与失效机制，保证读取性能和变更后可见性。

**Non-Goals:**
- 不在本次设计中替换系统内所有硬编码枚举，仅覆盖首批管理模块和可直接替换字段。
- 不引入独立配置中心或多环境字典发布流程。
- 不在本次设计中扩展国际化翻译能力。

## Decisions

### 1) 数据模型采用若依标准双表
- 决策：新增 `sys_dict_type`（类型定义）和 `sys_dict_data`（类型项）两表，`dict_type` 为业务唯一键，`sys_dict_data.dict_type` 作为逻辑关联字段。
- 原因：与若依认知一致，便于后续迁移、对照文档和团队复用经验。
- 备选：单表 JSON 存储。未选原因：难做单项权限治理、排序和状态过滤，查询与维护成本更高。

### 2) 管理接口与消费接口分离
- 决策：管理端提供完整 CRUD；消费端提供“按 `dictType` 拉取启用项”的轻量接口。
- 原因：管理端与业务消费端关注点不同，分离后更易做缓存和限权。
- 备选：统一复用管理列表接口。未选原因：返回冗余字段、性能和安全边界不理想。

### 3) 缓存粒度按 `dictType`
- 决策：缓存键按字典类型维度维护（如 `sys_dict:<dictType>`），字典类型或数据变更时精准清理对应键。
- 原因：命中率与失效成本平衡，避免全量清缓存导致抖动。
- 备选：全量字典单键缓存。未选原因：任一类型变更都会触发全量失效。

### 4) 前端采用“页面管理 + 业务按类型消费”
- 决策：新增字典类型/字典数据管理页面；业务页面通过统一 API 或 composable 按 `dictType` 取值。
- 原因：符合若依管理习惯，同时避免各页面重复请求和转换逻辑。
- 备选：仅提供后端接口，不做通用前端封装。未选原因：会重复造轮子并引入展示不一致风险。

### 5) 权限模型沿用现有 `system:*:*`
- 决策：新增菜单“字典管理”，按钮权限点采用 `system:dict:list|add|edit|remove`。
- 原因：对齐当前系统权限表达式和前端 `v-hasPermi` 使用方式。
- 备选：拆分为 `dictType`/`dictData` 两套权限点。未选原因：首期复杂度高，收益不足。

## Risks / Trade-offs

- 风险：历史页面仍有硬编码，短期内“字典化覆盖率”不完整  
  → 缓解：在任务中明确首批替换清单，并在后续迭代逐步扩展。
- 风险：缓存失效遗漏导致管理变更后前台短暂不一致  
  → 缓解：类型与数据写操作统一走服务层失效入口，并补充集成测试。
- 取舍：首期保持权限点粗粒度（`system:dict:*`）  
  → 后续如需更细粒度，可在不破坏主模型前提下扩展。

## Migration Plan

1. 数据库新增字典表与索引，初始化基础字典类型和示例数据。  
2. 后端上线字典管理与消费接口，并增加权限点与菜单初始化。  
3. 前端上线字典管理页面与统一字典获取能力。  
4. 首批页面替换硬编码枚举为字典数据来源。  
5. 灰度观察后清理冗余常量，保留兼容兜底。  

回滚策略：接口和页面按模块可回滚；数据库新增表保留不删除，仅停用入口和菜单。
