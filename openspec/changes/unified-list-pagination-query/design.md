## Context

前端已完成 feature-based 拆分，但列表页实现风格仍不统一。当前需求强调：
- 统一页面结构和按钮布局；
- 统一查询交互（查询/重置/展开收起）；
- 统一分页能力；
- 避免页面初始化即查询，改为用户显式触发。

## Goals / Non-Goals

**Goals**
- 四类页面采用同一套列表骨架。
- 分页能力由后端接口原生支持，前端不做全量后切片。
- 保持权限码控制逻辑不变，仅调整布局与数据获取方式。

**Non-Goals**
- 不改动业务主模型（company / role / menu / dict）。
- 不在本次引入高级筛选缓存、列配置持久化等增强能力。

## Decisions

### 1. 统一分页协议
后端列表接口统一返回分页对象：
- `records`: 当前页数据
- `total`: 总数
- `pageNum`: 当前页
- `pageSize`: 每页条数

前端统一请求参数：
- `pageNum`
- `pageSize`
- 各页面自有筛选字段

### 2. 统一列表布局
每个页面统一分为：
1) 查询区（可折叠）
2) 工具栏（左：新增/导入/导出，右：查询/重置/展开收起）
3) 表格
4) 分页器

### 3. 查询触发策略
- 页面初始不自动拉列表。
- 用户点击“查询”后请求第一页。
- “重置”清空筛选并可选触发第一页查询（本次采用：重置后不自动查，保持“点击查询才查数据”一致）。

### 4. 展开/收起
- 默认收起（仅显示第一行关键筛选项）。
- 点击“展开/收起”切换高级筛选项显示。

## Risks / Trade-offs

- 风险：从自动查询改为手动查询后，部分用户感知“页面初始为空”。
  - 对策：表格空态文案提示“请设置条件后点击查询”。
- 风险：分页接口改造涉及前后端联动，回归面增大。
  - 对策：先统一分页响应结构，再逐页切换前端消费。
