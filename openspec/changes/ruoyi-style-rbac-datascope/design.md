## Context

现有系统状态：
- 后端仅有 `sys_user` 与基础 JWT 校验，过滤器中权限固定为 `ROLE_USER`。
- 前端仅基于 token 做路由拦截，无菜单树、无按钮权限控制。
- 查询层无数据权限约束机制。

目标是引入“若依式”统一权限框架，但保持与当前技术栈兼容（Spring Security + MyBatis-Plus + Vue3）。

## Goals / Non-Goals

**Goals:**
- 建立标准组织与权限主数据模型，支持用户-角色-菜单按钮-岗位-部门。
- 提供统一接口鉴权能力：`hasPermi`、`hasRole`。
- 提供统一数据权限能力：按角色 `data_scope` 自动注入查询条件。
- 支持前端动态菜单与按钮显隐，并与后端鉴权保持一致。
- 明确分阶段实施与验收标准，保障可执行性。

**Non-Goals:**
- 不在本变更中实现完整审计中心（仅预留日志扩展点）。
- 不在本变更中引入多租户模型。
- 不在本变更中覆盖所有业务模块，仅先覆盖用户与权限管理链路。

## Decisions

### 1. 领域模型决策
- 采用若依常见模型：
  - `sys_dept`（树结构，含 `ancestors`）
  - `sys_post`
  - `sys_role`（含 `role_key`、`data_scope`）
  - `sys_menu`（`M/C/F`：目录/菜单/按钮）
  - 关联表：`sys_user_role`、`sys_role_menu`、`sys_user_post`、`sys_role_dept`
- `sys_user` 增加 `dept_id`、`del_flag` 等治理字段。

### 2. 鉴权决策
- JWT 仅承载轻量身份（`userId`、`username`、`tokenVersion`），权限实时从缓存/DB 装载。
- 接口权限统一采用 `@PreAuthorize("@ss.hasPermi('xxx')")`。
- 超级管理员通过 `role_key=admin` 兜底放行（保留审计日志）。

### 3. 数据权限决策
- 角色 `data_scope` 采用五档：
  - `1` 全部
  - `2` 自定义部门
  - `3` 本部门
  - `4` 本部门及以下
  - `5` 仅本人
- 采用 `@DataScope(deptAlias, userAlias)` + AOP 注入查询条件，优先保证可读性与快速接入。

### 4. 前后端契约决策
- 登录后返回用户基础信息与权限摘要。
- 新增权限接口：
  - 当前用户路由树
  - 当前用户按钮权限集合
- 前端动态注册路由；按钮通过 `v-hasPermi` 控制显隐；后端鉴权作为最终防线。

### 5. 实施顺序决策
- 阶段 1：主数据建模与基础 RBAC。
- 阶段 2：数据权限。
- 阶段 3：前端动态权限与管理页面联动。
- 阶段 4：缓存失效与审计增强。

## Risks / Trade-offs

- 风险：`@DataScope` SQL 拼接如果约束不足，可能产生条件遗漏或绕过。
  - 对策：统一 Mapper 入参对象、禁止手写绕过查询、增加集成测试矩阵。
- 风险：权限缓存与角色变更一致性。
  - 对策：角色/菜单/用户授权变更后按 `userId` 精确失效。
- 取舍：JWT 不携带完整权限，换取权限变更即时生效与令牌体积可控。
- 取舍：先保证若依核心模型落地，不一次性覆盖全部历史接口。
